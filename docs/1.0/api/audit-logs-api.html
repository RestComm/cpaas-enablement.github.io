Audit Logs

Introduction

Audit Logs (a.k.a Audit Events) allows you to view a log of activity within your Restcomm Account including all the Solutions developed on top of Restcomm, for example Message Exchange for Cisco UC-One, Message Exchange for Cisco WebEx Teams, Auto Attendant etc. This log contains detailed information about actions taken by users within the Restcomm or Solutions Dashboard or changes made by your application via the API.

For example provisioning a Phone Number or Visual Designer Application or editing the URL against Phone Number etc are all logged.

This information can be used by your security/compliance team to understand who made what changes to Restcomm Resources and also help for debugging.

Audit logs will only record logs for resources that are created, updated or deleted. If it’s a simple read operation, activity is not logged.

User can retrieve the Audit Log via REST API call as explained in Audit Log API Definition.

Logs and Events are interchanged in definition below, but they mean one and the same.
Audit Logs are available only for the last 6 months after which its deleted and cannot be retrieved.

Audit Log API Definition

Base Resource URI

https://your_custom_domain.com

Get Audit Logs

/auditlogs

Supported Operations

Http GET: list of event for provisioning action on the RC platform

Filter

Below are supported parameters for filtering event.

Parameter

Descriptions

organizationSid

Only include audit log event for this specific organization SID, this parameter is available if the user is Telestax super admin account.

accountSid

Only include auditlog event for this specific accountSid, this parameter is available for telestax super admin account and master account.

actorSid

Only include audit log event for this specific actor Sid.

eventType

Only include audit log event for this specific eventType

resourceSid

Only include audit log event for this specific resource Sid

sourceIpAddress

Only include audit log event for this specific source IP address.

limit

Limit the number of audit logs events that can be retrieved in one request.

Response will carry nextToken attribute which can be used to query next page of audit logs for this current request.

nextToken

To query next page of audit log event for previous request.

startTime

The start of the time range, expressed as the number of milliseconds after Jan 1, 1970 00:00:00 UTC. Events with a timestamp before this time are not returned.

endTime

The end of the time range, expressed as the number of milliseconds after Jan 1, 1970 00:00:00 UTC. Events with a timestamp later than this time are not returned.

Response Body

Response will be HTTP 200 OK along with JASON payload carrying events details if all ok. Else corresponding error will be returned.

The response returned is JSON body as shown in the example below. Each parameters are as explained below

Parameter

Description

sid

Unique Event Id

organization_sid

Unique SID of Organization to which this event belongs

account_sid

Restcomm Account SID that created the resource.

actor_sid

SID of Actor that caused the event.

For example if Enterprise Account created PhoneNumber, account_sid will be returning the SID of Enterprise Account and if CSP Admin made changes to this number, actor_sid will be the Account SID of CSP.

actor_type

Not used for now and will always be either ACCOUNT, USER or ADMIN for now. TODO

description

Description of event. Can be null.

event_data

Additional data about the event. The contents depend on event_type. For example, event-types of the form RESOURCE.updated, this value contains a resource attributes that describes the previous and updated properties of the resource.

event_date

Date and time when event was recorded. Date format is given as an ISO-8601 date/time string. Note that the given date/time is inclusive and is assumed to be in UTC timezone

event_type

The event’s type. Event-types are typically in the form: RESOURCE_TYPE.ACTION, where RESOURCE_TYPE is the type of resource that was affected (see resource_type parameter below) and ACTION is what happened to it. For example, phone-number.created. For a full list of all event-types, see the List of supported Resource Type and Event Type

links

The relative URL of Actor and Resource

resource_sid

The SID of resource that was affected

resource_type

The type of resource that was affected. For a full list of all resource-types, see the List of supported Resource Type and Event Type

source

The originating system or interface that caused the event. Can be: web for events caused by user action in the Restcomm Console, api for events caused by a request to our API.

source_ip_address

The IP Address of Source

uri

Relative URL of this event.
