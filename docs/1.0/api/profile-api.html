Profiles

Overview

Restcomm Profiles allow you to associate behavior configuration with specific Account or Organization. Once a Profile is associated with such Entity, the next session (Voice/SMS) attempt related to that Entity will use the Profile configuration to tweak the service logic behavior.

The Profile API allows different types of profiles to be provisioned.

The Network Profile is managed by the CSP administrator and is used for configuring behaviour like call & SMS routing with Bring Your Own Carrier (BYOC) and Media Session Control. The Network Profile type is Subscriber.

The different kinds of profiles are denoted by Content-Type header by setting proper schema parameters. Where "rc-subscriber-schema.json" corresponds to Subscriber profile.

During session creation the service logic will try to find the associated Network Profile based on the criteria below, until a Profile is found:

  • Profile associated with triggering Account

  • Profile associated with corresponding Organization

  • Default Subscriber Profile

This also means that Accounts and Organizations are always effectively associated with one Network Profile even if you havent set one for them explicitly. Typically those are the Default ones until you override those by configuring more suitable ones that take effect through the 'inheritance' process described above.

You can check which Network Profiles, Accounts and Organizations are associated with by retrieving them through their respective API Endpoints and inspecting the Link Response Headers.

Access Control

For Subscriber profile, the Profiles REST API is only accessible by a CSP administrator account. Where the created Profiles will only be accessible by the corresponding CSP account.

Default Profiles

The Default Network Profile is pre-provisioned in the system, and cant be modified from the REST API. The Default Network Profile Sid is "PR2a61d1839442571eb2eb88262064d1f9"

Since the Profile configuration is expected to grow as required, the Profile entity is defined by a JSON schema, rather than regular properties, representing a document used as a generic container.

The REST API will validate well-known contents of the document that are used by Restcomm during session handling logic. The rest of the document will not be validated, with the only constraint of a maximum size limitation of 10MB. This means that the REST API consumer may use the Profile document to store/retrieve any kind of arbitrary information as long as the document is a valid JSON document.

Resource Schemas

For more information about resource schemas details, please have a look at https://<ORG>.restcomm.com/restcomm/2012-04-24/Profiles/schemas/rc-subscriber-schema.json

Create Profile

POST /restcomm/2012-04-24/Profiles

Creates a new Profile and returns the representation. The body of the request is expected to contain a JSON document compliant to the Profile schema.

Content-Type HTTP header needs to be passed specifying that json should be provided along with the schema for the created profile. Below is the values of the Content-Type:

  • Subscriber Profile: application/instance+json;schema="<ENDPOINT>/restcomm/2012-04-24/Profiles/schemas/rc-subscriber-schema.json".

You can check Resource Schemas Details for more details on the various Profile schemas.

After successful creation the profileSid of the new Profile is returned in the Location response header.

Subscriber Profiles can only be created by the CSP Administrators.

Establishing media connection when the remote node is behind NAT (White list of nodes IP addresses, SIP inbound)

This functionality helps you enhance your security levels by whitelisting a range of IP addresses allowed to communicate with Restcomm. The feature is disabled by default. It can be enabled when needed by adding the "mediaRemoteWhitelist" parameter to the "sessionControl" subschema of the Subscriber profile. You need to specify one or more IP addresses that and only that will be allowed to initiate connections under this profile. IP addresses may be present in a form of a single address (like 192.168.1.1) or as a range (like 192.168.2.0/24).

{
    "sessionControl": {
        "mediaRemoteWhitelist": [
            "192.168.1.1", "192.168.2.0/24"
        ]
    }
}

Outbound SIP Routing

Enabling DNS resolution

By default the usage of DNS resolution for calls is disabled. To enable it you need to add an "dns_resolve" addition of the "multiProvider.outbound-voice.proxy.uri" Network Profile parameter.

Example

{
    "multiProvider": {
        "outbound-voice": [
            {
                ....
                "proxy": [
                    {
                        "uri": "10.0.0.1:5060;dns_resolve",
                        ....
                    }
                ]
            }
        ],
    },
}

Manipulation of SIP fields

This session control extension allows fine-grained control on how DNS resolution shall affect different fields and headers of the outbound SIP requests.

{
    "sessionControl": {
        "resolveRequestUri": "resolve(default)|keep",
        "resolveToHeader" : "resolve(default)|keep"
    }
}

The resolveRequestUri decides whether the hostname in the SIP Request URI shall be replaced with outbound-voice proxy defined in the MultiProvider extension. If the value is set to keep, no resolution is attempted and the field is left unmodified.

The resolveToHeader decides whether the hostname in the SIP To header shall be replaced with outbound-voice proxy defined in the MultiProvider extension. If the value is set to keep, no resolution is attempted and the field is left unmodified.

If not specified, both extensions default to resolve to maintain backwards compatibility.

Mapping of billing headers

The billing headers are sent in SIP INVITE / MESSAGE requests and responses as SIP custom headers, and in SMPP messages as TLV fields. "P-Asserted-Service" header is used for identifying which service is originating the traffic (e.g. Visual Designer, 2FA, Number Masking, Auto Attendant etc.) for charging/billing purposes. Extra headers can be assigned per an enterprise account. Renaming of headers names is possible by adding a "headerTranslationMap" into a subscribe profile. This can be specified separately for voice and messages.

Replacement example for voice:
{"multiProvider": {
  "outbound-voice":[
    {
      "proxy":[
        {
          "uri": "replacement.com",
          "username": "yourusername",
          "password": "yourpassword",
          "headerTranslationMap": {
            "P-Asserted-Service": "X-SomeCustomHeader"
          },
        },
      ]
    }
  ]
}
Replacement example for messages:
{"multiProvider": {
  "outbound-sms": [
    {
      "destination_network_id":"101",
      "headerTranslationMap": {
        "P-Asserted-Service":"X-TLV-COCTET-3000", "X-MSIDN":"X-TLV-COCTET-3001"
      }
    }
  ]
}}

When the headerTranslationMap is applied the "P-Asserted-Service" will be replaced with "X-SomeCustomHeader" for voice, and "X-TLV-COCTET-3000" for messaging (if that is a SIP message that reaches a customer).

Translated headers with names that start with "X-TLV-" have a special meaning. Headers with such names will be converted into SMPP TLVs before being sent by the SMPP protocol.

The name has the following format: "X-TLV-<TLV type>-<TLV tag (a decimal value)>". The table below contains a list of possible TLV types.

Update Profile

PUT /restcomm/2012-04-24/Profiles/{profileSid}

Modifies a Profile resource and returns the representation. The body of the request is expected to contain a JSON document compliant to the Profile schema. Similar to creating a profile, during profile modification you need to pass Content-Type HTTP header and specify proper content type and schema.

Profile updates can be organized into two categories:

  • Updates that affect the document of the actual Profile, like updating the BYOC section of a subscriber profile, so that another Network Profile is used (described in this section)

  • Updates that link/unlink the profile to an Account or Organization in order to configure the service logic behavior for them (described in subsection Link/Unlink Profile to Account/Organization)

Subscriber Profiles can only be modified by the CSP Administrators that created them.

Response Status

Status

Description

200 OK

Modification performed. Body contains Profile JSon document

403 Forbidden

Only CSP Admin is allowed to modify a Profile. DefaultProfile cant be modified

404 Not Found

Target Profile SID is missing

400 Bad Request

Body contains a Profile not compliant with schema

Link/Unlink Profile to Account/Organization

Profile association operation may be performed against a particular Account or Organization. This operation will use HTTP 1.1 LINK/UNLINK methods, and the target resource should be populated in a Link header, using "related" as "rel" attribute. Only one Link header is supported. To associate a single Profile with multiple Accounts/Organizations, multiple requests will be required.

Request Headers

Header

Description

X-Provisioning-Operation

Specifies if this is a LINK or UNLINK request

Link

Contains an full URL to the Account/Organization that should be associated to the Profile

Response Status

Status

Description

200 OK

Association established

403 Forbidden

Only CSP Admin is allowed for the association operation

404 Not Found

Target Profile SID is missing or Link is not found

400 Bad Request

None or multiple Link header provided. Link with unsupported "rel".

Examples

Link a Profile to an Account

curl -X PUT  https://<accountSid>:<authToken>@yourorganization.com/restcomm/2012-04-24/Profiles/PRae6e420f425248d6a26948c17a9e2123 -H "X-Provisioning-Operation:LINK" -H "Link:<https://<accountSid>:<authToken>@yourorganization.com/restcomm/2012-04-24/Accounts/ACae6e420f425248d6a26948c17a9e2acf>;rel=related"

Get Specific Account Relationship details

GET /restcomm/2012-04-24/Profiles/{profileSid}/relationships/{accountSid}

Retrieve a specific Account relationship to a Profile.

Access to reading a Profiles specific relationships is subject to the same rules as when reading the Profile itself

Response Attributes

Attribute

Description

account

Full details of the Account linked to the Profile

target_sid

The target of the relationship i.e. the AccountSid linked to the Profile

source_sid

The source of the relationship: it can be an OrganizationSid (if the account inherits it from the Org), an Account SID (if inherited from a Parent account) or a Profile SID (if using the default Profile).

profile_sid

The ProfileSid of the Profile whose relationships we are listing

Examples

Get details for the relationship of an Account to a Profile

GET https://<accountSid>:<authToken>@yourorganization.com/restcomm/2012-04-24/Profiles/PRebdc5dd54a5347f9ae0a6ed00512b934/relationships/AC5c80b0a09d5cf87c4394539c8ebaa48f

Delete Profile

DELETE /restcomm/2012-04-24/Profiles/{profileSid}

Removes an existing Profile, removing any existing association with Accounts/Organizations.

Subscriber Profiles can only be deleted by the CSP Administrator who created them except the Default Subscriber Profile (i.e. PR2a61d1839442571eb2eb88262064d1f9) that cannot be deleted by anyone.

Response Status

Status

Description

200 OK

Profile removed

403 Forbidden

Only CSP Admin is allowed to remove. DefaultProfile cant be deleted

404 Not Found

Target Profile SID is missing or Link is not found

Examples

Delete a Subscriber Profile

To delete a Subscriber Profile, run the command below.

curl -X DELETE https://<accountSid>:<authToken>@yourorganization.com/restcomm/2012-04-24/Profiles/PRae6e425f4f5218d3a2194ac1aa9ea6f1

Distributed Request Processing

The Restcomm platform has a distributed microservice architecture, that enables linear scalability during traffic growth and handles a high volume of REST API requests, voice calls, and SMS messages. Each architecture component handles different types of requests.

When you modify the provisioning data via the Management REST APIs, up to few minutes are needed for the update to reflect the changes in all call processing units that handle calls and SMS. Provisioning data modifications are done using the HTTP POST, PUT and DELETE methods. The POST method updates records immediately with no delay during the operation.

Distributed request processing is available for the following Management APIs, as well as the Restcomm Console.

  • Accounts

  • Tags

  • Profiles

  • Clients

  • Incoming Phone Numbers
