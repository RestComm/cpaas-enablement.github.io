Best Practices for Application Development

Checking Resources Status

Restcomm Resources such as Calls and Messages have status that tracks their real state in the network. Applications may depend on this status changes to invoke their own logic. The recommendation is to always use callback notifications for call/message events instead of polling the API.

Callback notifications are used for announcing states of calls/messages (call completed, message sent, failed etc). This allows reporting immediately after an event has occurred.

Usage of the Restcomm REST API (like https:/<name>@yourcompany.com/restcomm/2012-04-24/Accounts/<AccountSID>/SMS/Messages/<MessageID>) or WEB console for checking the state of a call/message does not guarantee that an information is available immediately after a call/message state is reached and it also consumes extra resources at the cloud services side.

URI Opacity

URIs provided by the Restcomm API are not meant to be parsed or scanned in any way. The URIs conform to HTTP URI specification, that is the only assumption a Developer may make.

As an Example Resources Sid are part of the URI. As such they should not be parsed. Do not rely on a format of Sids. Telestax can not guarantee that identifiers formats will not be changed in some time in future. The recommendation is not to check identifiers length or format at a customer application side.

The same is also valid for SIP headers like Call-ID. Restcomm only honors SIP specification in terms of format.

URI Discovery

When interacting with the Restcomm API to create a Restcomm Application, try to prevent URI composition and URI hardcoding on the application side as much as possible. Instead, try to follow the links provided by Resource representations, instead of trying to infer the convention and hardcoding that is in your application code.

For example a Restcomm Recording exposes the corresponding WAV file through an URI that is based in the original Recording URI. Instead of trying to compose the WAV file URI on you application code, just get the link for the Recording resource attribute 'file_uri'.
